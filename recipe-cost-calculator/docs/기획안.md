# 레시피 원가계산기 (Recipe Cost Calculator) 기획안

## 1. 개요

음식점·카페·베이커리 사장님 또는 개인이 **메뉴(레시피)의 원가를 정확히 계산**하고, 목표 마진에 맞는 **판매가를 산정**할 수 있도록 돕는 웹 애플리케이션.

- **한 줄 정의**: 재료비를 입력하면 메뉴 한 그릇(1인분)의 원가와 적정 판매가를 자동 계산해 주는 도구
- **플랫폼**: 웹 (반응형, 모바일 우선)
- **기술 스택**: React 19 + TypeScript + Vite, 상태는 로컬(localStorage) 우선

## 2. 타깃 사용자 & 문제 정의

| 사용자 | 겪는 문제 | 본 앱의 해결 |
| --- | --- | --- |
| 소규모 외식업 사장 | 감으로 가격 책정 → 마진 손해 | 정확한 재료비 기반 원가 산출 |
| 베이커리/홈베이킹 셀러 | g 단위 환산 계산이 번거로움 | 구매 단위 ↔ 사용 단위 자동 환산 |
| 예비 창업자 | 메뉴별 수익성 비교 어려움 | 메뉴별 원가율·마진 한눈에 비교 |

## 3. 핵심 개념 (도메인 모델)

```
재료(Ingredient)  ──┐
                    ├──> 레시피(Recipe) ──> 원가계산(Costing) ──> 판매가/마진
사용량(amount)   ──┘
```

### 3.1 Ingredient (재료)
- `id`, `name` (예: 밀가루)
- `purchasePrice` (구매가, 예: 10,000원)
- `purchaseAmount` + `unit` (구매 용량, 예: 1000 g)
- → **단가** = 구매가 / 구매용량 (예: 10원/g) *자동 계산*

### 3.2 Recipe (레시피/메뉴)
- `id`, `name` (예: 크루아상)
- `items[]`: { ingredientId, amount, unit } 사용 재료 목록
- `yield` (생산량/인분, 예: 10개)
- (선택) `lossRate` 손실율, `laborCost` 인건비, `overheadCost` 기타 경비

### 3.3 Costing (원가 계산 결과 - 파생값)
- **총 재료비** = Σ(재료 단가 × 사용량)
- **1인분 원가** = 총원가 / yield
- **원가율(%)** = 원가 / 판매가 × 100
- **마진/이익** = 판매가 − 원가
- **권장 판매가** = 원가 / (1 − 목표원가율) ← 목표 마진 역산

## 4. 기능 범위

### MVP (1차 — 반드시)
1. **재료 관리**: 등록/수정/삭제, 구매단위 기준 단가 자동 계산
2. **레시피 빌더**: 재료 선택 + 사용량 입력, 단위 자동 환산
3. **원가 계산**: 총원가, 1인분 원가 실시간 표시
4. **판매가/마진**: 판매가 입력 시 원가율·이익 표시 / 목표 마진 입력 시 권장가 역산
5. **데이터 저장**: localStorage 영속화 (새로고침해도 유지)

### 2차 (선택 — 여유 시)
6. 손실율·인건비·부대비용 반영한 정밀 원가
7. 메뉴별 수익성 **비교 대시보드** (원가율 정렬)
8. 데이터 **가져오기/내보내기** (JSON/CSV)
9. 단위 그룹 환산 (무게 g↔kg, 부피 ml↔L, 개수)

### 향후 (Backlog)
- 다중 사용자/클라우드 동기화 (백엔드 + DB)
- 재료 시세 변동 기록·알림
- PDF 원가표 출력

## 6. 기술 설계

- **상태 관리**: React Context + `useReducer` (또는 Zustand 도입 검토). 외부 상태 라이브러리 없이 시작
- **영속성**: `useLocalStorage` 커스텀 훅으로 상태 ↔ localStorage 동기화
- **타입**: `src/types/` 에 도메인 타입 정의 (Ingredient, Recipe 등)
- **계산 로직 분리**: `src/lib/costing.ts` 순수 함수로 작성 → 단위 테스트 용이
- **단위 환산**: `src/lib/units.ts` (무게/부피/개수 그룹)
- **폴더 구조(안)**
  ```
  src/
  ├─ types/        도메인 타입
  ├─ lib/          costing.ts, units.ts (순수 로직)
  ├─ hooks/        useLocalStorage, useStore
  ├─ store/        Context + reducer
  ├─ components/   재사용 UI
  └─ pages/        Ingredients, Recipes, Dashboard
  ```

## 7. 데이터 모델 (TypeScript 초안)

```ts
type Unit = 'g' | 'kg' | 'ml' | 'L' | 'ea';

interface Ingredient {
  id: string;
  name: string;
  purchasePrice: number;   // 원
  purchaseAmount: number;  // 구매 용량
  unit: Unit;
}

interface RecipeItem {
  ingredientId: string;
  amount: number;
  unit: Unit;
}

interface Recipe {
  id: string;
  name: string;
  items: RecipeItem[];
  yield: number;           // 생산량(인분/개수)
  sellingPrice?: number;   // 판매가
  targetCostRate?: number; // 목표 원가율(%)
}
```

## 10. 확정된 결정 사항

| 항목 | 결정 | 비고 |
| --- | --- | --- |
| 원가 범위 | **재료비만 (MVP)** | 손실율·인건비·경비는 2차 백로그로 이동 |
| 스타일링 | **Tailwind CSS** | 반응형 UI 빠르게 구성, 의존성 추가 필요 |
| 데이터 저장 | **localStorage 우선 → 추후 백엔드 동기화** | 저장 계층을 추상화해 추후 교체 용이하게 설계 |

### 결정에 따른 설계 영향
- **저장소 추상화**: `src/store/` 에 `StorageAdapter` 인터페이스를 두고, 우선 `LocalStorageAdapter` 구현 → 나중에 `ApiAdapter`로 무중단 교체. 컴포넌트는 저장 위치를 알 필요 없음.
- **Tailwind 도입**: `tailwindcss` + `@tailwindcss/vite` 플러그인 설치, `index.css`에 디렉티브 추가.
- **ID/타임스탬프**: 백엔드 전환 대비 `id`는 `crypto.randomUUID()`, `createdAt/updatedAt` 필드를 처음부터 포함.

## 12. 상세 기능 명세

### F-1. 재료 관리
| 항목 | 내용 |
| --- | --- |
| 입력 필드 | 이름(필수), 구매가(필수, >0), 용량(필수, >0), 단위 |
| 자동 계산 | 단가 = 구매가 ÷ 용량, 단위당 원가 표기 (예: 10원/g) |
| 유효성 | 빈 이름·0 이하 값 차단, 중복 이름 경고(허용은 함) |
| 삭제 제약 | **레시피에서 사용 중인 재료 삭제 시 경고** (해당 레시피 목록 표시) |
| 수정 | 인라인 또는 모달 편집, 수정 시 관련 레시피 원가 자동 재계산 |

### F-2. 레시피 빌더
| 항목 | 내용 |
| --- | --- |
| 재료 추가 | 드롭다운에서 기존 재료 선택 → 사용량·단위 입력 |
| 단위 환산 | 재료 단위와 사용 단위가 다르면 자동 환산 (g↔kg 등 동일 그룹 내) |
| 단위 불일치 | 호환 불가 단위(g↔ml 등) 선택 시 경고 표시 |
| 생산량 | yield(인분/개수), 1 이상 |
| 실시간 | 입력 변경 시 원가 패널 즉시 갱신 (디바운스 불필요, 순수 계산) |

### F-3. 원가/가격 계산
| 출력 | 공식 |
| --- | --- |
| 총 재료비 | Σ (재료 단가 × 환산 사용량) |
| 1개당 원가 | 총 재료비 ÷ 생산량 |
| 원가율(%) | (1개당 원가 ÷ 판매가) × 100 |
| 1개당 이익 | 판매가 − 1개당 원가 |
| 권장 판매가 | 1개당 원가 ÷ (1 − 목표원가율/100) |

> 판매가 미입력 시 원가율·이익은 "—" 표시. 0으로 나누기 방어 처리.

### F-4. 데이터 영속화
- 모든 변경은 `StorageAdapter`를 통해 저장 → 현재는 localStorage 키 `rcc:ingredients`, `rcc:recipes`
- 앱 시작 시 로드, 변경 시 자동 저장(디바운스 300ms)
- (2차) JSON 내보내기/가져오기로 백업

## 13. 계산 예시 (검증용)

**재료 단가**
- 밀가루: 10,000원 ÷ 1,000g = **10원/g**
- 버터: 8,000원 ÷ 500g = **16원/g**
- 설탕: 3,000원 ÷ 1,000g = **3원/g**

**크루아상 레시피 (생산량 10개)**
| 재료 | 사용량 | 단가 | 비용 |
| --- | --- | --- | --- |
| 밀가루 | 500g | 10원/g | 5,000원 |
| 버터 | 200g | 16원/g | 3,200원 |
| 설탕 | 50g | 3원/g | 150원 |
| **합계** | | | **8,350원** |

- 1개당 원가 = 8,350 ÷ 10 = **835원**
- 판매가 2,000원일 때 원가율 = 835 ÷ 2,000 = **41.75%**, 이익 = **1,165원**
- 목표 원가율 30% → 권장 판매가 = 835 ÷ 0.7 = **약 1,193원** ... (목표가 더 낮은 원가율이면 가격 ↑)

> 위 수치는 구현 후 단위 테스트(`costing.test.ts`)의 기준값으로 사용.

## 15. 엣지 케이스 & 검증 규칙
- 0 또는 음수 입력 차단 (구매가, 용량, 사용량, 생산량)
- 판매가 0/미입력 시 원가율 계산 회피 (NaN/Infinity 방지)
- 호환 불가 단위 환산 시도 시 명확한 에러 메시지
- 사용 중인 재료 삭제 시 영향받는 레시피 안내
- localStorage 용량 초과/파싱 실패 시 graceful 처리(초기화 옵션)
- 숫자 표시: 원화 반올림(원 단위), 천 단위 콤마
